Complex App Delivery Board

Mission: Build a complex app from idea to shipping using looping Claude agents.

Loop command: /loop 1m ./docs/workboard.html ./docs/idea.md

Context

Product idea: A collaborative planning app for small software teams.

Primary users: founders, designers, engineers, QA.

Constraints: clear documentation, incremental delivery, release readiness.

Operating Rules

  1. Always claim a ready unowned task before making changes.
  2. Do not start blocked work.
  3. After completing a task, update statuses for newly unblocked tasks.
  4. Evaluation tasks must run 5 passes minimum.

Task Board

D1 - Build initial HTML mocks

Goal: Create the first set of HTML mocks for the main product flows.

Inputs: idea.md

Deliverable: HTML mock files checked into the repo.

Parallel: Can run alone. Unlocks design evaluation.

Done when: core screens exist and are reviewable.

D2 - Evaluate HTML mocks for 5 passes

Goal: Review and improve the initial mock set over five iterations.

Inputs: D1 outputs, idea.md

Deliverable: revised HTML mocks plus recorded pass count.

Parallel: No downstream task starts until complete.

Done when: passes 1 through 5 are complete and final mocks are accepted.

R1 - Create requirements from idea and approved design

Goal: Produce product requirements using the idea doc, design outputs, and technical guidance docs.

Inputs: idea.md, approved mocks, technical guidance docs

Deliverable: requirements document in the repo.

Parallel: Can unlock architecture and test-planning tasks later.

Done when: scope, features, constraints, and acceptance criteria are documented.

R2 - Evaluate requirements for 5 passes

Goal: Review and improve requirements over five iterations.

Inputs: R1 outputs, idea.md, mocks

Deliverable: approved requirements document.

Parallel: Must finish before implementation planning.

Done when: all five passes are completed and open issues are resolved.

Q1 - Prepare repo for shipping

Goal: Create README, CI/CD workflow, lint setup, test plan, test execution, and release checklist.

Inputs: approved requirements and implemented features

Deliverable: repo is documented, validated, and ready to ship.

Parallel: Split into sub-tasks where no file ownership conflicts exist.

Done when: docs, CI/CD, linting, tests, and release checks are complete.

Decision Log

  • Use evaluation gates after major deliverables.
  • Prefer small tasks that a single loop can finish.
  • Keep dependency data current so multiple agents can coordinate safely.
