App Delivery Workflow

Audience: Claude agent only. This document is the single source of truth for what to do next. Do not rewrite it for human readability.

Mission: Drive ./docs/idea.md to a deployed app using this fixed, evaluable, parallelizable workflow.

Loop command: /loop 1m ./workflow.html ./docs/idea.md

Agent Protocol (run this every loop iteration)

Multi-agent coordination: Two or more agents may be running this workflow on different machines. workflow.html in origin/main is the shared lock table. Every status or owner change MUST be committed and pushed immediately so the other agent sees it on its next iteration. Identify yourself with a stable, unique data-owner value — for example claude@<hostname>. Never use the bare value claude when more than one agent is active.

  1. Sync from remote first. Run git fetch origin then git pull --rebase origin main in the repo root. Do not read workflow.html until the pull succeeds. If the working tree is dirty (uncommitted edits from a prior aborted iteration), git stash them, pull, then git stash pop and resolve before continuing.
  2. Read this file and parse every <article data-task-id="..."> element.
  3. Resolve statuses: for any task whose data-status="blocked", recheck dependencies. If every ID in data-depends-on has data-status="done" (or the list is empty), update that task's data-status to ready. If you flipped any statuses, save, commit (chore(workflow): unblock <ids>), and push before continuing.
  4. Pick the next task using this priority order:
    1. Any task already data-status="in-progress" with data-owner equal to your identifier — resume it.
    2. Otherwise, the first task in document order with data-status="ready" and data-owner="".
    3. If no ready task exists and any task is still blocked or in-progress by another owner, exit this iteration (wait for the next loop tick).
    4. If every task has data-status="done", the workflow is complete — write WORKFLOW COMPLETE and stop the loop.
  5. Claim it atomically (this is the critical section for two-agent safety):
    1. Set data-status="in-progress" and data-owner="<your-id>" on the chosen article. Save.
    2. git add workflow.html
    3. git commit -m "claim(workflow): <task-id> by <your-id>"
    4. git push origin main
    5. If the push is rejected (non-fast-forward), the other agent claimed something concurrently. git reset --hard origin/main after fetching, then restart this iteration from step 1. Do not retry the same task without re-reading.
    The push-success is the lock acquisition. Until it succeeds, you do not own the task.
  6. Execute the task per its Goal, Inputs, and Deliverable. Use the skill named in data-skill if one exists; otherwise proceed without a skill. Commit and push deliverable artifacts (code, docs, tests) as you produce them, on their own commits separate from workflow.html state changes.
  7. Evaluation tasks (data-evaluation-passes > 0): run the evaluation that many times, appending findings to ./docs/evaluations/<task-id>.md before marking done. Implementation tasks use ATDD: write the acceptance test first, then make it pass.
  8. On completion:
    1. Set data-status="done" on the article and clear data-owner.
    2. Re-run step 3 so newly unblocked tasks flip to ready in the same edit.
    3. Save, then git add workflow.html, git commit -m "done(workflow): <task-id>", git push origin main.
    4. If push is rejected: git pull --rebase origin main, reapply your status change on top (your done still wins because you held the lock), then push again.
  9. If blocked mid-task, leave data-status="in-progress", append a note to a sibling <p data-note="blocker"> inside the article describing what is needed. Save, commit (note(workflow): <task-id> blocker), push, and exit the iteration. The other agent must not steal an in-progress task; only the owner clears it.
  10. End of iteration: ensure your working tree is clean and your local main matches origin/main. If not, you have unpushed work — push it before the loop fires again.

Concurrency rules:

  • Tasks sharing a data-parallel-group may proceed independently once dependencies are met. With two agents, the second one picks the next ready task in document order — usually the parallel sibling.
  • Only the owner of an in-progress task may modify its data-status or data-owner. Never reassign someone else's task.
  • If you find a task in-progress by another owner that has had no commits in 30 minutes (check git log -1 --format=%cI -- workflow.html), append a <p data-note="stale-owner"> note but do not claim it. A human will resolve.
  • Push-rejection on a workflow.html commit is the expected concurrency signal — handle it, do not panic. Push-rejection on a deliverable commit is a normal merge — pull-rebase and re-push.

Never delete tasks, reorder articles, modify data-task-id, force-push main, or change human-readable wording outside the data-* attributes and the data-note elements you append.

Attribute Legend

  • data-task-id: stable identifier referenced by data-depends-on.
  • data-status: ready | blocked | in-progress | done.
  • data-owner: unique agent identifier currently holding the task (e.g. claude@laptop-01); empty when unclaimed.
  • data-parallel-group: tasks sharing a group can run together when their dependencies are met.
  • data-depends-on: comma-separated task IDs that must be done before this task starts.
  • data-skill: the Claude skill to invoke for this task (or empty for none).
  • data-evaluation-passes: required number of evaluation iterations (0 = not an evaluation task).

Task Board

D1 - Create HTML mocks

Goal: Produce HTML mocks of the core product flows.

Inputs: idea.md

Deliverable: HTML mock files committed to the repo.

Done when: core screens exist and are reviewable.

D2 - Evaluate HTML mocks

Goal: Review mocks against idea and improve until accepted.

Inputs: D1 outputs, idea.md

Deliverable: revised, accepted mocks; review notes recorded.

Done when: all blocking findings resolved.

R1 - Create requirements

Goal: Translate idea + accepted mocks into product requirements.

Inputs: idea.md, accepted mocks

Deliverable: requirements document in the repo.

Done when: scope, features, constraints, and acceptance criteria are written.

R2 - Evaluate requirements

Goal: Review and refine requirements until accepted.

Inputs: R1 outputs, idea.md, mocks

Deliverable: approved requirements document.

Done when: all blocking findings resolved.

MB1 - Create backend MVP

Goal: Build a thin backend MVP that proves the architectural pattern.

Inputs: approved requirements

Deliverable: running backend MVP with documented patterns (project layout, conventions, sample endpoint).

Parallel: Runs in parallel with MF1.

Done when: MVP runs locally and patterns are documented.

MB2 - Evaluate backend MVP

Goal: Validate backend MVP patterns against requirements.

Inputs: MB1 outputs, requirements

Deliverable: accepted MVP and pattern reference.

Done when: all blocking findings resolved.

MF1 - Create frontend MVP

Goal: Build a thin frontend MVP that proves the UI architecture and integration approach.

Inputs: approved requirements, accepted mocks

Deliverable: running frontend MVP with documented patterns (routing, state, component structure, sample page).

Parallel: Runs in parallel with MB1.

Done when: MVP runs locally and patterns are documented.

MF2 - Evaluate frontend MVP

Goal: Validate frontend MVP patterns against requirements and mocks.

Inputs: MF1 outputs, requirements, mocks

Deliverable: accepted MVP and pattern reference.

Done when: all blocking findings resolved.

BP1 - Plan backend implementation

Goal: Plan backend implementation grounded in requirements and MVP patterns.

Inputs: approved requirements, accepted backend MVP

Deliverable: implementation plan (modules, data model, API surface, sequencing).

Done when: plan covers every requirement.

BP2 - Evaluate backend plan

Goal: Review backend plan for completeness, risk, and consistency with MVP patterns.

Inputs: BP1 outputs

Deliverable: approved backend plan.

Done when: all blocking findings resolved.

BT1 - Create vertically sliced backend tasks

Goal: Break the backend plan into small, vertically sliced tasks.

Inputs: approved backend plan

Deliverable: task list; each task ships an end-to-end slice.

Done when: tasks cover the plan and each is small enough to finish in one loop.

BT2 - Evaluate backend tasks

Goal: Review backend task list for slice quality, sizing, and coverage.

Inputs: BT1 outputs

Deliverable: approved backend task list.

Done when: all blocking findings resolved.

BI1 - Implement each backend task with ATDD

Goal: For every approved backend task: write the acceptance test, implement until green, evaluate, then move to the next.

Inputs: approved backend task list

Deliverable: task slice merged with passing acceptance test and evaluation note.

Per-task loop: implement (ATDD) -> evaluate implementation -> mark task done.

Done when: every backend task is implemented, evaluated, and merged.

FP1 - Plan frontend implementation

Goal: Plan frontend implementation grounded in requirements and MVP patterns.

Inputs: approved requirements, accepted frontend MVP

Deliverable: implementation plan (routes, components, state, integration points).

Done when: plan covers every requirement and mock screen.

FP2 - Evaluate frontend plan

Goal: Review frontend plan for completeness and consistency with MVP patterns.

Inputs: FP1 outputs

Deliverable: approved frontend plan.

Done when: all blocking findings resolved.

FT1 - Create vertically sliced frontend tasks

Goal: Break the frontend plan into small, vertically sliced tasks.

Inputs: approved frontend plan

Deliverable: task list; each task ships an end-to-end UI slice.

Done when: tasks cover the plan and each is small enough to finish in one loop.

FT2 - Evaluate frontend tasks

Goal: Review frontend task list for slice quality, sizing, and coverage.

Inputs: FT1 outputs

Deliverable: approved frontend task list.

Done when: all blocking findings resolved.

FI1 - Implement each frontend task with ATDD (Playwright POM)

Goal: For every approved frontend task: write the Playwright acceptance test using the Page Object Model, implement until green, evaluate, then move to the next.

Inputs: approved frontend task list

Deliverable: task slice merged with passing Playwright acceptance test (POM) and evaluation note.

Per-task loop: implement (ATDD with Playwright POM) -> evaluate implementation -> mark task done.

Done when: every frontend task is implemented, evaluated, and merged.

DP1 - Create Azure deployment workflow (cheapest plan)

Goal: Add a Git workflow that deploys the app to Azure on the cheapest viable plan, provisioned via the Azure CLI.

Inputs: implemented backend and frontend

Deliverable: CI/CD workflow file, Azure CLI provisioning script, and deployment docs.

Done when: a push to the deploy branch provisions and deploys successfully.

TP1 - Create test plan

Goal: Author a test plan covering happy paths, edge cases, and regressions.

Inputs: requirements, implemented app

Deliverable: test plan document in the repo.

Done when: all requirements have at least one test scenario.

TP2 - Ensure app runs locally

Goal: Verify backend and frontend start cleanly on a fresh checkout.

Inputs: repo

Deliverable: documented local run command and a green smoke check.

Done when: a fresh clone runs the app from a single command.

TP3 - Run test plan 5 times, log bugs, fix bugs

Goal: Execute the test plan five times. Each pass: log bugs to a markdown file in ./docs/bugs/, then fix them before the next pass.

Inputs: test plan, running app

Deliverable: 5 dated bug logs under ./docs/bugs/ and fixes merged for each.

Per-pass loop: run plan -> append bugs to ./docs/bugs/pass-N.md -> fix -> rerun until pass N is clean -> proceed to pass N+1.

Done when: pass 5 finishes with no new blocking bugs.

Parallelism Map

  • mvp-backend (MB1, MB2) runs in parallel with mvp-frontend (MF1, MF2) once R2 is done.
  • backend-impl (BP1 -> BP2 -> BT1 -> BT2 -> BI1) runs in parallel with frontend-impl (FP1 -> FP2 -> FT1 -> FT2 -> FI1).
  • Within each implementation track, BI1 and FI1 iterate per task: implement (ATDD) -> evaluate -> next.
  • Deployment (DP1) waits for both BI1 and FI1.

Decision Log

  • Evaluation gates follow every produced artifact (mocks, requirements, MVPs, plans, task lists, implementations).
  • ATDD is mandatory for implementation tasks; Playwright Page Object Model is mandatory for frontend tasks.
  • MVPs exist to lock in patterns before scaling implementation.
  • Vertically sliced tasks are the unit of progress during implementation.
  • Azure deployment uses the cheapest viable plan provisioned via Azure CLI.
  • Test plan executes 5 full passes, with bugs tracked per-pass under ./docs/bugs/.
